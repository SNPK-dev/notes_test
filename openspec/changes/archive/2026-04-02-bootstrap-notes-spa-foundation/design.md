## Context

Нужен технический фундамент для Notes SPA на Nuxt 4, который позволит реализовывать сценарии работы с заметками без смешивания доменной логики, хранения данных и представления. Из proposal следуют три опорные capability: каркас приложения, persistence-контракт и UI foundation. Дополнительные ограничения важны уже на этапе дизайна: приложение должно иметь ровно две страницы, не использовать native alert/confirm, сохранять возможность заменить localStorage на REST API без переписывания UI и domain, а также оставаться SSR-safe по архитектурным границам, даже если runtime выбран как SPA.

Текущее состояние предполагает старт практически с нуля, поэтому дизайн должен сразу зафиксировать слои, точки расширения и границы ответственности. Отдельно важно развести persisted notes state и transient editor state: данные заметки должны жить в репозитории и store списка, а состояние редактирования, dirty-флаги, hydration-флаги и будущая history-модель должны жить отдельно от presentational components.

## Goals / Non-Goals

**Goals:**
- Определить слоистую структуру `domain -> repositories -> stores -> components/pages` с явными контрактами между слоями.
- Зафиксировать каркас маршрутов для двух страниц: списка заметок и единой страницы редактора заметки.
- Ввести repository abstraction для заметок с localStorage-адаптером, доступным только на клиенте.
- Разделить store списка заметок и store редактора, чтобы UI-компоненты оставались тонкими и переиспользуемыми.
- Зафиксировать persisted model, editor draft model и поток синхронизации между ними.
- Описать modal confirmation flow для потенциально опасных действий и базовые keyboard shortcuts, не завязанные на конкретный визуальный компонент.
- Подготовить архитектуру для будущих undo/redo без реализации полного history-механизма в этом change.
- Зафиксировать runtime validation при чтении данных из localStorage.
- Зафиксировать требования к Dockerfile и docker-compose.yaml как части foundation-поставки.

**Non-Goals:**
- Полная реализация всех CRUD-сценариев и всей бизнес-логики редактора в одном change.
- Полноценная реализация undo/redo history, включая команды, стек изменений и UI-индикаторы.
- Реализация backend, синхронизации между устройствами, аутентификации и offline-first поведения.
- Реализация rich-text, тегов, поиска, вложений и других расширенных возможностей заметок.
- Создание сложной дизайн-системы сверх минимально достаточного набора reusable UI-компонентов.
- Многоступенчатая container orchestration или инфраструктурные решения, не нужные для локального запуска тестового задания.

## Decisions

### 1. Слоистая структура приложения

Приложение будет разделено на четыре уровня ответственности:
- `domain`: типы, value objects, мапперы, валидаторы и чистые функции для заметок и todo-элементов.
- `repositories`: интерфейсы доступа к данным и конкретные адаптеры, например `LocalStorageNotesRepository`.
- `stores`: orchestration-слой на Pinia, который управляет загрузкой, сохранением, transient state и координацией UI-сценариев.
- `components/pages`: страницы, layout и переиспользуемые компоненты без прямого доступа к browser storage и без доменных побочных эффектов.

Физическая структура проекта будет следовать конвенциям Nuxt, а не абстрактному каталогу `ui`. То есть UI-слой раскладывается по `pages/`, `components/`, `layouts/`, `composables/`, а не складывается в одну отдельную папку `ui`.

Почему так: это соответствует заданным ограничениям по separation of concerns, не конфликтует с конвенциями Nuxt и делает замену persistence-слоя локальным изменением. Альтернатива с логикой прямо в компонентах или одном большом store быстрее на старте, но ломает SSR-safe границы, ухудшает тестируемость и затрудняет замену localStorage на API.

### 2. SPA runtime с SSR-safe границами

Приложение будет запускаться как SPA, чтобы соответствовать тестовому заданию. При этом архитектура не должна зависеть от прямого доступа к browser-only API в domain, stores и компонентах. Все обращения к `localStorage`, `window`, `document` и глобальным keyboard listeners должны быть изолированы в client-only infrastructure слое.

Почему так: задание требует SPA, но quality bar и требования вакансии подразумевают понимание SSR-safe проектирования. Поэтому runtime выбирается как SPA, а архитектурные границы строятся так, чтобы приложение можно было эволюционно адаптировать под SSR или HTTP-backed persistence без переписывания UI и domain.

### 3. Ровно две page-компоненты и единая страница редактора

Приложение использует ровно две page-компоненты:
- `pages/index.vue` — список заметок;
- `pages/editor/[[id]].vue` или эквивалентную единую editor page — создание новой заметки и редактирование существующей.

Создание и редактирование работают в одном editor route с разным режимом:
- отсутствие `id` означает создание новой заметки;
- наличие `id` означает редактирование существующей.

Почему так: это напрямую соответствует ограничению тестового задания на две страницы и предотвращает расползание маршрутов на `new`, `edit`, `details` и вспомогательные формы.

### 4. Два Pinia store вместо одного универсального

Будут выделены как минимум два store:
- `useNotesStore` для persisted state, списка заметок, загрузки, удаления и синхронизации с репозиторием;
- `useNoteEditorStore` для transient editor state: текущего draft, dirty/isSaving/isHydrated флагов, производных селекторов, save/cancel логики и будущей history-обвязки.

Почему так: список заметок и редактор имеют разный жизненный цикл. Persisted state должен быть общим и долговечным, а editor state может пересоздаваться при навигации и сбросе. Альтернатива с одним store приводит к смешиванию долгоживущих данных и локального состояния формы, усложняя confirm-on-leave, cancel и будущий undo/redo.

### 5. Persisted state и transient editor state разделены явно

`useNotesStore` хранит только persistable данные и состояние загрузки коллекции. `useNoteEditorStore` работает с копией заметки в виде draft-модели, которая может отличаться от persistable модели в процессе редактирования.

Будут зафиксированы как минимум две модели:
- persisted model — сериализуемая и стабильная структура, которую читает и пишет repository;
- editor draft model — transient-представление, пригодное для редактирования и будущей history-модели.

Почему так: сохранённая версия заметки и редактируемый draft — это разные сущности с разным жизненным циклом. Альтернатива, где форма редактирует persisted state напрямую, ломает cancel, undo/redo и контроль несохранённых изменений.

### 6. Save/cancel flow принадлежит editor store

Editor store будет владельцем потока редактирования:
- при открытии редактора store создаёт draft на основе существующей заметки или дефолтной модели новой заметки;
- при изменениях обновляется только draft;
- `save` валидирует draft, преобразует его в persistable model, вызывает repository и синхронизирует `useNotesStore`;
- `cancel` отбрасывает transient editor state и не меняет persisted state;
- dirty-флаг вычисляется относительно исходного snapshot/editor baseline.

Почему так: save/cancel — это не UI-детали формы, а часть сценария редактирования. Если размазать эту логику по компонентам, появится дублирование, сложность тестирования и ошибки синхронизации между списком и редактором.

### 7. Undo/redo как обязанность editor store, а не UI

Хотя полный undo/redo не входит в текущий scope, дизайн сразу резервирует для него место внутри `useNoteEditorStore`. История изменений будет относиться к transient editor state и мыслиться как стек snapshots или набор editor-команд. UI-компоненты только вызывают действия store, например `updateTitle`, `toggleTodo`, `requestUndo`, не зная о внутреннем представлении истории.

Почему так: undo/redo относится к поведению редактора, а не к отображению. Если привязать историю к компонентам формы, появится дублирование логики, слабая тестируемость и риск несовместимости при замене UI-компонентов.

### 8. Repository abstraction с SSR-safe client adapter

Будет введён интерфейс наподобие `NotesRepository` с методами чтения и записи заметок. Реализация на localStorage будет располагаться отдельно и использоваться только на клиенте через Nuxt plugin, factory или composable-провайдер, который не обращается к browser-only API на сервере.

Пример контракта уровня ответственности:
- `getAll(): Promise<Note[]>`
- `getById(id: string): Promise<Note | null>`
- `save(note: Note): Promise<Note>`
- `remove(id: string): Promise<void>`

Почему так: localStorage недоступен во время SSR, а прямой доступ из stores или компонентов создаёт хрупкость. Альтернатива с `process.client` или `import.meta.client` проверками по всему приложению распределяет инфраструктурную ответственность по разным слоям и затрудняет будущую замену на HTTP-репозиторий.

### 9. Runtime validation для данных из localStorage обязательна

Данные, прочитанные из localStorage, считаются недоверенным runtime payload и должны проходить валидацию в repository layer до преобразования в доменную модель. При невалидных или повреждённых данных репозиторий возвращает безопасный fallback, а не пробрасывает сырой payload дальше в stores и UI.

Рекомендуемая реализация:
- DTO-схема для сериализуемых данных;
- runtime validation через `zod` или эквивалентный валидатор;
- безопасный fallback на пустую коллекцию или пропуск повреждённых записей с логированием.

Почему так: TypeScript не валидирует данные во время выполнения, а localStorage может содержать устаревший, повреждённый или вручную изменённый payload.

### 10. Нормализованная доменная модель заметки

В `domain` будет зафиксирована минимальная стабильная модель заметки и todo-элемента.

Базовый контракт:
```ts
type Todo = {
  id: string
  text: string
  completed: boolean
}

type Note = {
  id: string
  title: string
  todos: Todo[]
  createdAt: string
  updatedAt: string
}
````

В persisted model не должны попадать UI-поля вроде `isDirty`, `isModalOpen`, `isSelected`, `isEditing` и другие transient-флаги. Editor store строит поверх persisted model отдельное transient-представление, пригодное для редактирования.

Почему так: persisted model должна оставаться стабильной и независимой от текущего состояния интерфейса. Это упрощает миграции, сериализацию и замену persistence-слоя.

### 11. Hydration flow управляется notes store

Hydration списка заметок происходит только на клиенте. До завершения hydration приложение не должно пытаться читать browser storage. `useNotesStore` должен иметь флаг гидрации, например `isHydrated`, чтобы UI мог различать:

* начальное безопасное состояние;
* процесс загрузки клиентских данных;
* гидратированное состояние с доступной коллекцией заметок.

Editor store не должен самостоятельно читать localStorage напрямую. Он получает исходные данные через repository и/или `useNotesStore`.

Почему так: это предотвращает чтение browser-only API в неподходящий момент и делает клиентскую инициализацию предсказуемой.

### 12. Modal confirmation flow через централизованный controller и единый host

Подтверждения удаления, ухода со страницы с несохранёнными изменениями и других опасных действий будут выполняться не через native alert/confirm, а через централизованный modal controller. Компоненты и stores инициируют запрос подтверждения через унифицированный API, например `requestConfirmation(...)`, а единый global modal host отображает диалог.

Планируемый поток:

1. Store, composable или страница вызывает `requestConfirmation(...)`.
2. Modal controller публикует состояние активного диалога.
3. Global modal host отображает диалог с фокус-ловушкой, заголовком, описанием и действиями `confirm/cancel`.
4. Результат возвращается инициатору в виде promise/result.
5. Инициатор решает, выполнять удаление, навигацию, discard draft или отменять сценарий.

Почему так: такой подход выполняет UX-требование, не дублирует confirm-диалоги по приложению и сохраняет единый контроль над фокусом, escape-handling и жизненным циклом подтверждения.

### 13. Keyboard shortcuts и accessibility как часть store-driven UX

Foundation должен поддержать клавиатурные действия без жёсткой привязки к DOM-конкретике:

* `Esc` закрывает активную modal или инициирует cancel editor-сценария;
* `Ctrl/Cmd + S` триггерит сохранение из editor store, если форма валидна и есть изменения;
* `Ctrl/Cmd + Z` и `Shift + Ctrl/Cmd + Z` резервируются под будущие undo/redo действия editor store;
* навигация и базовые controls должны оставаться доступными через стандартный tab order и видимые focus states.

Границы действия:

* сочетания `Ctrl/Cmd + S`, `Ctrl/Cmd + Z`, `Shift + Ctrl/Cmd + Z` активны только на editor page;
* глобальные listeners не должны перехватывать сочетания вне контекста редактора;
* обработчики keyboard shortcuts должны подключаться и отключаться централизованно, а не разрозненно по компонентам.

Почему так: требования доступности должны быть встроены в foundation, а не добавляться постфактум. Альтернатива с разрозненными `window.addEventListener` в компонентах приведёт к конфликтам обработчиков и слабой тестируемости.

### 14. Docker как часть deliverable, но в минимальной конфигурации

`Dockerfile` и `docker-compose.yaml` рассматриваются как часть foundation и должны обеспечивать предсказуемый локальный запуск приложения в production-like режиме. При этом containerization остаётся минимальной:

* один application service;
* без декоративной multi-service оркестрации;
* без лишнего infrastructure overhead, не требуемого заданием.

Допустимая цель:

* `docker-compose up` поднимает работоспособную копию приложения;
* контейнер собирает и запускает Nuxt-приложение предсказуемым способом;
* локальная проверка не требует ручных дополнительных шагов вне README/compose.

Почему так: в тестовом задании поставка артефактов равноправна коду приложения. Альтернатива "добавим контейнеризацию потом" создаёт риск, что архитектурные решения и скрипты запуска не будут согласованы.

## Risks / Trade-offs

* [Разделение на два store увеличивает количество файлов и координации] -> Компенсируется более ясными границами ответственности и упрощением editor-сценариев.
* [Repository abstraction может выглядеть избыточной для localStorage-only версии] -> Это осознанная плата за заменяемость persistence и соблюдение запрета на прямой доступ к storage из UI.
* [SSR-safe обвязка вокруг browser-only API добавляет инфраструктурный код] -> Она локализуется в plugin/factory-слое, чтобы остальная часть приложения оставалась простой.
* [Runtime validation добавляет дополнительный слой DTO и преобразований] -> Это снижает риск повреждённого localStorage payload и делает поведение приложения предсказуемым.
* [Централизованный modal controller требует аккуратной работы с фокусом и promise lifecycle] -> На foundation-этапе ограничивается одним типовым confirm-flow и единым host-компонентом.
* [Резервирование keyboard shortcuts до полной реализации сценариев может создать ложное ощущение готовности] -> В specs и tasks должно быть явно указано, какие сочетания пока только зарезервированы.
* [Включение Docker в foundation может расширить первый change] -> Риск компенсируется минимальной конфигурацией контейнеризации без лишней инфраструктурной сложности.

## Migration Plan

1. Создать каркас Nuxt-приложения и слоёв `domain`, `repositories`, `stores`, `components/pages`, не нарушая конвенции Nuxt.
2. Зафиксировать доменные типы `Note` и `Todo`, DTO-схемы и runtime validation для persisted payload.
3. Подключить `useNotesStore` и `useNoteEditorStore` с разделением persisted state и editor draft state.
4. Добавить `NotesRepository` contract, SSR-safe localStorage adapter и точку его внедрения в клиентский runtime.
5. Подготовить app shell, routing skeleton, global modal host и базовые reusable UI-компоненты.
6. Подготовить keyboard shortcuts infrastructure для editor page без полной undo/redo реализации.
7. Добавить Dockerfile и docker-compose.yaml, затем проверить production-сборку и локальный запуск.
8. Если реализация foundation окажется неработоспособной, rollback выполняется откатом change целиком, так как он не требует миграции пользовательских данных и не меняет внешний API.

## Open Questions
